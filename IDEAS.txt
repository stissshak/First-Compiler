MPL — Language design ideas / notes
====================================

Intentional deviations from C (keep these):
- One declarator per declaration. No `int a, b, c;`. Avoids C's
  `int* a, b` footgun. parseVarDecl already enforces this.
- Structs auto-register as type names. After `struct Foo {…}` you write
  `Foo x;`, not `struct Foo x;`. More ergonomic than C. (userTypes list
  in the parser, isType checks it.)
- Named parameters required. `int f(int)` is illegal; must be `int f(int x)`.

Ideas to explore:
- Statement as function body (not only a block).
  FuncDecl::body is already `unique_ptr<Stmt>`, so the AST supports it;
  only the parser restricts it via parseBlock(). Switching to parseStmt()
  would allow `int f() return 5;`.
  CAUTION: this collides with prototypes. A bare `;` after the param list
  must mean ONE thing. Decision: `;` = prototype (no body). Any non-empty
  statement = body. Never treat `;` as an "empty-statement body", or we
  lose prototype syntax.

- Prototypes / declarations without a body (BEING ADDED).
  `int printf(char*);`  -> FuncDecl with body == nullptr.
  Enables: calling external code (libc), separate compilation,
  forward/mutual recursion across the file.
  Codegen: body == nullptr -> emit `extern name` (or nothing), no definition.

TODO / gaps (not intentional — just not done yet):
- Array declarations: parseType only handles base type + `*`. No `[N]`
  parsing, so `int a[10]` can't be declared even though ArrayType exists
  in the AST. Need to add `[size]` handling in the type/declarator parser.
- Prototype + definition matching: declaring a prototype and later defining
  the same function currently errors ("Func was declarated") because the
  analyzer rejects any duplicate name. To support `int f(); ... int f(){}`
  the analyzer must allow a definition to "complete" a prior prototype,
  checking the signatures match instead of erroring.
